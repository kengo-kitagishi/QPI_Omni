# ヒストグラムビン数の最適化

## 現在の設定

```python
hist_min = -1.1  # ヒストグラムの最小値（rad）
hist_max = 1.5   # ヒストグラムの最大値（rad）
n_bins = 512     # ヒストグラムのビン数
```

- 範囲: 2.6 rad
- ビン幅: 2.6 / 512 = **0.0051 rad/bin**

## ユーザーの指摘（正しい）

### ナイキスト限界の考察

1. **元画像のサンプリング限界**
   - ピクセルサイズで決まる空間分解能
   - それ以上細かい情報は**原理的に存在しない**

2. **ヒストグラムのビン幅**
   - 元画像の位相分解能より細かくしても意味がない
   - むしろノイズを増やすだけ

3. **統計的な考察**
   - ヒストグラムは画像全体のピクセルをビンに分配
   - 画像が例えば512x512 = 262,144ピクセル
   - 512ビン → 1ビンあたり平均512ピクセル
   - これは適切か？

## 適切なビン数の計算

### 方法1: スタージェスの公式

```
ビン数 = 1 + log2(N)
N = ピクセル数
```

例: 512x512 = 262,144ピクセル
```
ビン数 = 1 + log2(262144) = 1 + 18 = 19
```

→ **19ビンで十分！**（現在の512は過剰）

### 方法2: スコットの公式

```
ビン幅 = 3.5 * σ / N^(1/3)
σ = 標準偏差
```

バックグラウンドのσを0.1 radと仮定:
```
ビン幅 = 3.5 * 0.1 / 262144^(1/3) = 0.0054 rad
ビン数 = 2.6 / 0.0054 = 481
```

→ 約**500ビン**（現在の512に近い）

### 方法3: 物理的限界から

元画像の位相分解能を考える：
- 測定ノイズ: ~0.01 rad程度
- ガウスフィットの目的: バックグラウンドピークの位置（0.001 rad精度で十分）
- 必要なビン幅: 0.01 ~ 0.05 rad程度

```
ビン幅 = 0.02 rad の場合
ビン数 = 2.6 / 0.02 = 130
```

→ **100〜200ビンで十分**

## 推奨設定

### オプション1: 実用的（推奨）
```python
n_bins = 128  # または 256
```

理由：
- ビン幅: 2.6 / 128 = 0.02 rad
- ガウスフィットに十分な分解能
- スムージングとの相性が良い
- 計算も速い

### オプション2: 保守的
```python
n_bins = 256
```

理由：
- ビン幅: 2.6 / 256 = 0.01 rad
- より細かい構造も捉えられる
- 現在の512よりは軽い

### オプション3: ミニマル
```python
n_bins = 64
```

理由：
- ビン幅: 2.6 / 64 = 0.04 rad
- バックグラウンドピーク検出には十分
- 最も高速

## smooth_windowとの関係

```python
smooth_window = 20  # スムージングのウィンドウサイズ
```

- 現在: 512ビンで20ウィンドウ = 約4%をスムージング
- 128ビンで20ウィンドウ = 約16%をスムージング

推奨：
```python
n_bins = 128
smooth_window = 10  # ビン数に応じて調整
```

## ユーザーの「元画像のサンプリング」の解釈

おそらく意図：
- **画像のピクセル数程度のビン数で十分**
- 512x512画像なら、512ビンは妥当
- ただし、ナイキスト限界を考えると**もっと粗くても良い**

## 結論

**n_bins = 128** または **256** を推奨

理由：
1. ガウスフィットに十分な分解能
2. 計算が速い
3. ノイズの影響が小さい
4. 物理的に意味のある精度
