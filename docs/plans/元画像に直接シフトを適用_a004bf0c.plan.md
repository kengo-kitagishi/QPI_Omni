# 元画像に直接シフトを適用

## 問題

現在の処理：
1. 元画像（float64）→ uint8に変換
2. uint8でECC計算 → シフト量を取得
3. **uint8画像を平行移動** ← ここが問題
4. float64に戻す際に範囲が変わる → コントラスト変化

## 正しい処理（ユーザー指摘）

1. 元画像（float64）→ uint8に変換
2. uint8でECC計算 → シフト量を取得
3. **元のbg_corr画像（float64）を直接平行移動** ← これが正解！

## 実装

### 変更箇所

[`21_calc_alignment.py`](c:\Users\QPI\Documents\QPI_omni\scripts\21_calc_alignment.py) **228-235行**

#### 現在のコード
```python
# 変換適用
h, w = target_img.shape
aligned_uint8 = cv2.warpAffine(
    target_uint8, warp_matrix, (w, h),  # uint8を移動
    flags=cv2.INTER_LINEAR + cv2.WARP_INVERSE_MAP
)
aligned_img = aligned_uint8.astype(np.float64) / 255.0
aligned_img = aligned_img * (np.max(target_img) - np.min(target_img)) + np.min(target_img)
```

#### 修正後
```python
# 変換適用（元画像を直接移動）
h, w = target_img.shape
aligned_img = cv2.warpAffine(
    target_img.astype(np.float32),  # 元画像を直接移動
    warp_matrix, 
    (w, h),
    flags=cv2.INTER_LINEAR + cv2.WARP_INVERSE_MAP
).astype(np.float64)
```

**変更点**:
- `target_uint8`ではなく`target_img`を使用
- uint8→float64の変換処理を削除
- シンプルに元画像を平行移動

## メリット

1. ✅ **コントラストが完全に保存** - 元の値がそのまま保たれる
2. ✅ **シンプル** - 余計な変換がない
3. ✅ **正確** - 値の範囲が変わらない
4. ✅ **uint8は計算にのみ使用** - 本来の目的通り

## 理由

- uint8への変換は**ECCアルゴリズムのため**だけ
- 計算結果（シフト量）は元画像にも適用できる
- わざわざuint8を移動させる必要はない
