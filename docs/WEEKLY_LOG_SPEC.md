# 週次ログ書き方仕様

週次レポートの正本。この仕様に従ってレポートを書く。

## 1. 構成原則

- **トピック（内容）でまとめる**：例「compute_pos_shifts の閾値調整」「pipeline のバグ修正」
- 同じファイルが複数日に変更されていても、同じトピックなら1つにまとめる
- セッション境界で区切らない。「このセッション」「次のセッション」といった見出しは使わない
- 最終レポートの構成は**時間順ではなく内容（トピック）順**

## 2. 記述スタイル

- **因果の流れ**：こういう設計にした → 実行した → こういう図が出た → それを見て次にこう変更した → ... の順で自然につなぐ。図に関してはコードの変更点から、どういう変更をしてこの結果が出たのかを書くこと。
- **1つのトピックにつきQiita 相当の厚み**：1つのトピックにつき **最低 5000 日本語文字以上、目安 7000〜9000 字**。「5000字」は日本語文字カウントであり ASCII 換算ではない。もっと長くていい。背景・目的・設計意図・手順・コード抜粋・実行結果（定量値）・気づきをすべて含める。情報を落とさない。短くまとめようとしない。形式張った「まとめ」や PI open questions は不要。その分、各トピックの記述を厚くする
- **新規機能の設計説明** : 新しく実装した機能についてはコードの設計意図が誰が見ても読み取れるようにする。そのコードがおこなう計算や処理の内容やどのデータやどの段階のパイプラインに適用するのか、また物理的数学的な仮定はなんなのかがもしあれば含め、その実装によって得られる結果をコード抜粋と合わせて書く。読者がこのコードを理解して再現できるレベルを目指す。
- **解析パラメータ**: 閾値・ウィンドウサイズ等の具体的な値。変更した場合は前後を記録
- **データの出所**: 生データの識別（Pos、条件）。図のデータ来歴は要約でよい、図のリンクも載せる。図のデータに付随するnpz,csvファイルがある場合は図の下に載せる。
- **図の凡例・条件**: パネル内容・パラメータ要約・単位、横軸縦軸はデータの何を表しているか
- **見出し**：「まとめ」「結論」といった形式張った見出しは避ける。トピック名や作業内容を示す自然な見出しにする

## 3. Notion メモの扱い

- 索引に Notion メモの**全文**が含まれている。これを読み、「このユーザーは〇〇を考えている／〇〇を試したかった／〇〇が気になっている」と解釈して書き直す
- 要約や抜粋ではなく、意図が伝わる形でレポートに織り込む。

## 4. 因果推論の扱い

- 編集 timestamp（session_files / tool_calls）と図の run_id や `.figure_history/session_YYYY-MM-DD.json` の時刻を利用
- 同一日で「Edit X → Bash 実行 script X → Figure script X」の並びがあれば、その順序で因果を記述する
- claude_sessions .md のタイムライン（Edit/Bash の順）と figure inbox .md を突き合わせ、「同じ日・近い時刻の Bash（script X）と図（script X）」で因果を推論する

## 5. 禁止事項

- タスクリスト形式
- セッション単位の見出し（「Session A」「セッション 1」など）
- 薄いサマリのみ（「〇〇を修正した」だけで中身がない）

## 6. 品質チェック（Quality Gate）

- 各トピックに「設計 → 実行 → 結果（図）→ 次にやったこと」の流れがあるか
- コード・コマンドの抜粋があるか（再現できる程度）
- 定量的な記述または「解析中」などの明記があるか
- 図を適切に参照しているか（`![[filename]]` または `[fig_id vXXX]`）
- タスクリスト形式になっていないか
